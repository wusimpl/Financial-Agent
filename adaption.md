# 前端适配后端分阶段计划

说明：本计划基于当前前端页面和已完成的后端接口整理，目标是把前端从本地示例数据切换到真实后端数据。只覆盖前后端数据打通、页面状态和现有交互适配，不包含重新设计界面、账号体系、多用户能力或无关功能扩展。

当前状态：后端已经提供股票、自选股、行情图表、SEC 文件、财务数据、社媒观点和页面聚合接口；前端仍主要读取 `mockData.ts`，还没有真正请求后端。

## 适配原则

| 原则 | 说明 |
| --- | --- |
| 先打通主链路 | 优先让当前股票页面用 `/dashboard/{ticker}` 加载真实数据。 |
| 保持现有界面结构 | 不重做布局，不大改组件，只替换数据来源和必要状态。 |
| 集中做数据转换 | 后端返回结构和前端展示结构不完全一致，前端用单独适配层转换，避免组件里到处拼字段。 |
| 明确页面状态 | 加载中、正常为空、获取失败分别展示，避免真实数据失败时页面像坏了一样。 |
| 逐步移除假数据 | 真实接口覆盖到哪里，假数据就从哪里退出；不保留长期兜底假数据。 |

## 阶段一：接口访问基础

| 任务名称 | 任务描述 |
| --- | --- |
| 前端接口入口 | 新增统一的后端请求入口，避免组件直接写死请求地址。 |
| 本地开发代理 | 配置前端开发环境访问后端，保证浏览器能正常请求接口。 |
| 请求错误处理 | 统一处理接口失败、网络失败和后端错误信息。 |
| 类型对齐 | 根据后端返回结构补齐前端需要的数据类型。 |
| 数据适配层 | 新增后端响应到现有页面数据结构的转换函数。 |
| 基础联调检查 | 确认前端可以请求 `/health` 和 `/dashboard/AAPL`。 |

## 阶段二：首页主链路打通

| 任务名称 | 任务描述 |
| --- | --- |
| 当前股票数据加载 | 页面进入时根据当前股票代码请求聚合接口。 |
| 顶部股票信息 | 用真实数据展示股票代码、公司名、价格、涨跌额和涨跌幅。 |
| 交易状态 | 用后端返回的市场状态展示盘前、开盘、盘后、隔夜或休市。 |
| 行情图表 | 用真实 K 线、成交量、均线、RSI 和 MACD 替换示例图表。 |
| 财务数据 | 用真实历史财务数据替换财务表格中的示例数字。 |
| 社媒数据 | 用真实帖子列表替换示例帖子。 |
| 数据块状态 | 根据后端 `sources` 字段区分正常、空数据和获取失败。 |

## 阶段三：自选股和股票切换

| 任务名称 | 任务描述 |
| --- | --- |
| 自选股列表 | 用 `/stocks/watchlist` 获取侧边栏列表。 |
| 侧边栏展示 | 用真实股票代码和公司名称渲染侧边栏。 |
| 股票切换 | 点击自选股后重新请求对应股票的聚合数据。 |
| 切换加载状态 | 当前股票切换时展示明确加载状态，避免旧数据误导用户。 |
| 空列表状态 | 自选股为空时展示清晰的空状态。 |

## 阶段四：搜索能力适配

| 任务名称 | 任务描述 |
| --- | --- |
| 搜索接口接入 | 用 `/stocks/search` 支持股票代码和公司名称搜索。 |
| 搜索结果展示 | 展示股票代码、公司名称和匹配结果。 |
| 选择搜索结果 | 选择搜索结果后切换到对应股票页面。 |
| 搜索空状态 | 没有匹配股票时展示无结果状态。 |
| 搜索失败状态 | 搜索失败时展示可理解的错误信息。 |

## 阶段五：图表局部交互

| 任务名称 | 任务描述 |
| --- | --- |
| 时间范围切换 | 图表选择 1D、5D、1M、3M、6M、YTD、1Y、5Y、All 时请求对应数据。 |

| 任务名称 | 任务描述 |

## 阶段八：移除示例数据和收尾

| 任务名称 | 任务描述 |
| --- | --- |
| 移除首页假数据依赖 | 当前股票页面不再依赖 `mockData.ts`。 |
| 清理无用类型 | 删除适配完成后不再使用的示例数据类型。 |
| 清理无用文案 | 移除开发视角、示例数据相关或误导用户的页面文案。 |
| 前端类型检查 | 确认前端类型检查通过。 |
| 后端测试回归 | 确认后端现有测试通过。 |
| 浏览器验收 | 打开页面检查真实数据、股票切换、图表、财务和社媒显示正常。 |

## 推荐落地顺序

1. 先完成接口访问基础和 `/dashboard/{ticker}` 主链路打通。
2. 再接自选股和股票切换，让用户可以浏览多只股票。
3. 然后接搜索，补齐从搜索进入股票页面的路径。
4. 接着做图表、SEC、社媒这些局部刷新交互。
5. 最后清理 `mockData.ts` 依赖，做类型检查、后端测试和浏览器验收。

## 不建议当前阶段做的事

| 事项 | 不建议原因 |
| --- | --- |
| 重做前端布局 | 当前目标是数据打通，布局重做会扩大范围。 |
| 引入复杂状态管理 | 当前页面规模有限，先用现有 React 状态即可。 |
| 引入账号和权限 | 后端需求已经明确只做单用户。 |
| 长期保留假数据兜底 | 假数据会掩盖真实接口问题，适配完成后应移除。 |
| 一次性改完所有组件 | 风险大、难定位问题，应该按数据链路逐步替换。 |
