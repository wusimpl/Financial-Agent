# 后端需求清单

说明：本清单根据当前前端页面整理，只覆盖界面需要后端提供的数据和能力，不包含深浅色切换、面板展开、侧栏收起等纯界面交互。

只做单用户，不做多用户。

## 股票检索与自选股

| 需求名称 | 需求描述 |
| --- | --- |
| 股票搜索 | 支持按股票代码或公司名称搜索股票。 |
| 股票基础信息 | 返回股票代码、公司名称、公司标识等展示信息。 |
| 自选股列表 | 返回用户关注的股票列表及每只股票的基础信息。 |
| 当前股票切换 | 根据选中的股票代码返回对应页面数据。 |

## 个股概览

| 需求名称 | 需求描述 |
| --- | --- |
| 最新报价 | 返回当前价格、涨跌额和涨跌幅。 |
| 交易状态 | 返回盘前、开盘、盘后、隔夜、休市等市场状态。 |
| 日内行情 | 返回开盘价、最高价、最低价和昨收价。 |
| 成交数据 | 返回当前成交量和平均成交量。 |
| 估值指标 | 返回市值、市盈率、Beta、每股收益和目标价。 |
| 财务日历 | 返回下一次财报日期。 |
| 股息信息 | 返回下一次派息日、除息日和股息率。 |
| 价格区间 | 返回 52 周价格区间。 |

| 需求名称 | 需求描述 |
| --- | --- |
| 个股看板数据 | 一次返回当前股票页面所需的概览、图表、财务和社媒数据。 |

## 后端架构设计

### 设计边界

- 只支持单用户，不设计账号、权限、团队、租户等模块。
- 后端以“数据源接入 + 业务整理 + 页面聚合”为主，不提前引入复杂数据库。
- 现有 `MarketSource`、`SecFilingsSource`、`TwitterSource` 继续作为底层数据源类使用，上层新增服务类负责把原始数据整理成前端需要的结构。
- 页面聚合接口需要返回各数据块的获取状态。某个非关键数据块失败时，不应该让整个页面完全不可用，而是返回该数据块的失败状态和空列表。

### 模块总览

| 模块 | 主要职责 |
| --- | --- |
| API 路由模块 | 暴露股票搜索、自选股、个股概览、图表、SEC 文件、财务数据、社媒观点、页面聚合等接口。 |
| Schema 模块 | 定义接口输入输出结构，保证后端返回字段稳定。 ✅ |
| 应用服务模块 | 编排多个数据源，完成业务规则、数据清洗、数据组装。 |
| 数据源模块 | 对接行情、SEC、社媒等外部数据来源，只负责获取原始数据。 |
| 解析与计算模块 | 处理技术指标、SEC 正文解析、财务字段映射、市场状态判断等纯逻辑。 ✅ |
| 存储与缓存模块 | 管理单用户自选股、本地缓存和配置。 ✅ |
| 错误与状态模块 | 统一错误类型，并为页面聚合接口生成数据来源状态。 ✅ |

### API 路由模块

API 层建议使用 FastAPI 的 `APIRouter` 函数，不强行设计 Controller 类。原因是当前项目接口数量有限，服务类已经承担业务编排职责，再增加 Controller 类会让结构变重。

| 文件 | 路由职责 | 调用的服务类 |
| --- | --- | --- |
| `app/api/stocks.py` | 股票搜索、自选股列表、个股概览。 | `StockSearchService`、`WatchlistService`、`StockOverviewService` |
| `app/api/charts.py` | 按时间范围返回 K 线、成交量和技术指标。 | `ChartDataService` |
| `app/api/sec.py` | SEC 文件列表、文件正文、关键章节。 | `SecFilingService` |
| `app/api/financials.py` | 财务摘要和多年财务数据。 | `FinancialDataService` |
| `app/api/social.py` | 股票相关社媒帖子。 | `SocialPostService` |
| `app/api/dashboard.py` | 一次返回个股页面所需全部数据。 | `DashboardService` |

### 应用服务模块

服务类负责业务编排。它们可以调用多个数据源类和解析计算类，但不直接处理 HTTP 请求细节。

| 类 | 职责 |
| --- | --- |
| `StockSearchService` | 处理股票搜索。负责规范化搜索关键词，按股票代码或公司名称查找股票，并返回前端可展示的基础信息。 |
| `WatchlistService` | 管理单用户自选股列表。负责读取默认自选股、补齐股票基础信息，并返回稳定排序的列表。 |
| `StockOverviewService` | 组装个股概览。负责获取行情、公司资料、估值、股息、财务日历和 52 周区间，并整理为一个概览响应。 |
| `ChartDataService` | 组装图表数据。负责把前端时间范围转换成底层行情请求参数，获取 K 线数据，并补齐 MA、RSI、MACD 等指标。 |
| `SecFilingService` | 管理 SEC 文件能力。负责按代码、年份、文件类型筛选文件，获取正文，解析元信息和关键章节。 |
| `FinancialDataService` | 组装财务数据。负责从 SEC facts 中提取净销售额、销售成本、毛利、营业利润、净利润、每股收益等字段，并生成摘要和多年历史数据。 |
| `SocialPostService` | 组装社媒观点。负责按股票代码和公司名称构造查询条件，支持热门和最新排序，限制最多 50 条，并统一作者、正文、时间和互动数据格式。 |
| `DashboardService` | 组装个股看板。负责并行或顺序调用概览、图表、财务、社媒服务，返回页面一次加载需要的数据，并生成每个数据块的来源状态。 |

### 数据源模块

数据源类只负责“拿到原始数据”，不承担页面字段组装职责。

| 类 | 职责 |
| --- | --- |
| `MarketSource` | 获取美股行情原始数据，包括 K 线、成交量等。现有类可继续使用，后续只补充必要的报价和基础行情方法。 |
| `SecFilingsSource` | 获取 SEC 公司索引、文件列表、文件正文和 company facts。现有类可继续使用。 |
| `TwitterSource` | 获取社媒搜索结果和用户帖子。现有类可继续使用。 |
| `StockDirectorySource` | 提供股票目录查询能力。初期可以基于 SEC company tickers 实现，用于股票搜索和公司基础信息补齐。 |
| `MarketProfileSource` | 获取市值、市盈率、Beta、目标价、股息率等行情扩展资料。如果当前行情工具无法提供这些字段，该类负责隔离后续更换数据来源的影响。 |

### 解析与计算模块

这些类只处理确定性的转换、解析和计算，不直接请求外部数据。 ✅

| 类 | 职责 |
| --- | --- |
| `TickerNormalizer` | 统一股票代码格式，去空格、转大写，并校验空值。 ✅ |
| `MarketStatusResolver` | 根据交易所时间、交易日和当前时间判断盘前、开盘、盘后、隔夜、休市等状态。 ✅ |
| `ChartRangeMapper` | 把前端时间范围转换成底层行情需要的周期和数量，例如 1D 对应分钟级数据，1Y 对应日线数据。 ✅ |
| `TechnicalIndicatorCalculator` | 计算 MA20、MA50、MA200、RSI(14)、MACD、信号线和柱状值。 ✅ |
| `SecDocumentParser` | 清理 SEC HTML 或文本正文，生成可阅读正文。 ✅ |
| `SecSectionExtractor` | 从 SEC 正文中提取 Business、Risk Factors、MD&A 等关键章节。 ✅ |
| `SecMetadataMapper` | 从 SEC submissions 和 company facts 中整理文件类型、报告期、公司名称、档案编号、注册地和雇主识别号。 ✅ |
| `FinancialFactsMapper` | 把 SEC XBRL 标签映射到业务字段，例如净销售额、销售成本、毛利、营业利润、净利润和每股收益。 ✅ |
| `FinancialMetricCalculator` | 计算毛利率、收入增长等从原始财务数据派生出来的指标。当前需求未要求的指标不应提前返回。 ✅ |
| `SocialPostNormalizer` | 统一社媒帖子结构，整理作者、账号、头像标识、发布时间、相对时间和互动数据。 ✅ |

### 存储与缓存模块

当前只做单用户，优先使用本地文件和缓存，不需要数据库。 ✅

| 类 | 职责 |
| --- | --- |
| `Settings` | 管理后端配置，例如缓存目录、请求超时时间、SEC User-Agent、社媒浏览器设置等。现有类可继续使用。 ✅ |
| `FileCache` | 管理本地文件缓存，避免频繁请求外部数据源。现有类可继续使用。 ✅ |
| `WatchlistStore` | 保存单用户自选股代码列表。初期可使用配置文件或本地 JSON 文件。 ✅ |
| `CachePolicy` | 定义不同数据类型的缓存时间，例如行情、社媒、SEC 索引、SEC facts 等。配置项较少时可以先放在 `Settings` 中，不必单独实现。 ✅ |

### 错误与状态模块

错误模块需要让接口返回清晰结果，尤其是页面聚合接口要能区分“真的没有数据”和“数据获取失败”。 ✅

| 类 | 职责 |
| --- | --- |
| `AppError` | 应用层基础错误，表示业务处理失败。 ✅ |
| `NotFoundError` | 表示股票、SEC 文件或其他目标资源不存在。 ✅ |
| `SourceError` | 表示外部数据源请求失败。现有类可继续使用。 ✅ |
| `SourceStatusBuilder` | 把每个数据块的成功、失败、空数据情况整理成 `SourceState`，供页面聚合接口返回。 ✅ |
| `ApiErrorResponse` | 定义接口错误响应格式，保证错误信息结构稳定。 ✅ |

### 推荐落地顺序

1. 先补齐 Schema 模块，固定前后端数据契约。
2. 保留现有数据源类，新增应用服务模块，让接口不再直接返回原始数据源结果。
3. 优先实现 `DashboardService`，因为它能一次覆盖当前页面最主要的数据需求。
4. 再拆出股票搜索、图表、SEC、财务、社媒等独立接口，方便页面局部刷新和后续调试。
5. 最后补充 `SourceStatusBuilder`，让页面能明确展示正常为空和获取失败的区别。

### 不建议当前设计的模块

| 模块或类 | 不建议原因 |
| --- | --- |
| 用户账号模块 | 需求明确只做单用户，账号体系会增加无关复杂度。 |
| 权限模块 | 当前没有多角色和权限边界。 |
| 复杂任务队列 | 当前数据量小，页面请求触发即可；只有在外部数据请求明显变慢或需要后台预热时再考虑。 |
| ORM 数据模型 | 当前没有明确持久化业务数据，先用本地自选股文件和缓存即可。 |
| 通用 Provider 抽象层 | 目前只有少数数据源，过早抽象会增加代码量；等需要替换或并存多个行情供应商时再引入。 |
